# RTG Project Structure Notes

Короткая карта структуры проекта для быстрого входа в контекст.

## Корень репозитория

- `src/` — основной код приложения (CLI + TUI + слои домена/инфры/интеграций).
- `docs/` — публичные инженерные документы (roadmap, quality gate, runbook, security).
- `config.example.toml` — пример конфигурации.
- `README.md` — обзор проекта, запуск, базовые правила.
- `CONTRIBUTING.md` — правила вклада и ожидания по PR.

## Слои в `src/`

- `src/main.rs` — вход в приложение, регистрация модулей и запуск `app::run`.
- `src/app.rs` — композиция runtime-сценариев (`run`, `logout`) и маршрут startup flow.
- `src/ui/` — только взаимодействие с терминалом и рендер (`shell`, `view`, `event_source`, `terminal`, `state`).
- `src/domain/` — модели и состояния предметной области (`chat`, `status`, `events`, `shell_state`).
- `src/usecases/` — orchestration/usecase-логика (`startup`, `bootstrap`, `guided_auth`, `logout`, `list_chats`).
- `src/infra/` — адаптеры и инфраструктурные детали (`config`, `logging`, `storage_layout`, `secrets`, ошибки/контракты).
- `src/telegram/` — Telegram integration boundary (auth/connectivity/status tracking, адаптер).

## Логика разделения ответственности

- UI не содержит бизнес-правил: события идут в usecases, рендер строится из доменного состояния.
- Usecases оркестрируют сценарии и используют контракты, а не прямые инфраструктурные детали.
- Domain изолирован от UI/infra и хранит типы/состояния/события.
- Infra и Telegram — адаптерный слой: интеграции, хранение, конфиг, внешние зависимости.
- `infra/config/` выделен как отдельный подмодуль загрузки/валидации/адаптации конфигурации.

## Практический ориентир при изменениях

- Новую feature разбивать по слоям, не собирать все в одном файле.
- Валидацию, утилиты и типы выносить в отдельные модули.
- Держать модуль коротким (мягко: около 200 LOC) и с одной зоной ответственности.
- При добавлении важной навигационной информации обновлять `docs/internal/`.
